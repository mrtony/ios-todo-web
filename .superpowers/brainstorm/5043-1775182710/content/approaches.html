架構方案選擇

三個方案各有取捨，請選擇最適合你的方向

A

Monorepo 單一倉庫

結構：一個 GitHub repo，前後端分資料夾（/client + /server）

部署：前端 build 後由 Express 提供靜態檔案服務，整個 app 部署為單一服務（可用 Railway / Render / Fly.io 免費方案）

優點：

  • 部署最簡單 — 一個服務搞定前後端
  • SQLite 檔案就在同一台，零設定
  • GitHub 管理方便，一個 repo 一次 PR

缺點：

  • 無法用 GitHub Pages（需要後端運行環境）
  • 前後端耦合在同一個部署單元

⭐ 推薦方案 — 最符合 Express + SQLite 的特性

B

前後端分離部署

結構：一個 GitHub repo（monorepo），但前後端各自部署

部署：前端 → GitHub Pages，後端 → Railway / Render / Fly.io

優點：

  • 前端真的部署在 GitHub Pages 上
  • 前後端可以獨立擴展和部署

缺點：

  • 需要處理 CORS 跨域問題
  • 兩個部署流程要維護
  • 後端仍然需要額外的託管平台
C

Docker 容器化

結構：一個 GitHub repo，加上 Dockerfile 和 docker-compose

部署：Docker image 部署到任何支援容器的平台，或本機 docker compose up

優點：

  • 環境一致性最好，一鍵啟動
  • 未來擴展到團隊版時最靈活
  • 可部署到任何雲平台

缺點：

  • 初始設定較複雜
  • 本地開發需要安裝 Docker
  • 對個人版來說可能有點 overkill
